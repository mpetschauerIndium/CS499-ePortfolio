Enhancement Three

Database Enhancement Narrative

This narrative explains the MongoDB and Mongoose improvements made to strengthen persistence, validation, indexing, and data integrity.

Artifact Overview

The TRAVLR artifact stores trip inventory and administrative user records in MongoDB through Mongoose models. The database enhancement focused on making this persistence layer more reliable, secure, and maintainable.

Why the Artifact Was Selected

The application depends on accurate trip and user data, so the database layer was a strong place to demonstrate CS499 outcomes. Improving schemas, indexes, connection behavior, validation, and query safety directly improves the reliability of the full application.

Enhancement Goals

The goals were to define stricter Mongoose schemas, prevent unsupported fields from being persisted, add indexes for common lookup and sorting patterns, protect password material, control database connection behavior through configuration, and improve CRUD operations with validation-backed updates.

Analysis of the Original Artifact

The original artifact used MongoDB for trip data, but it needed stronger enforcement of data shape and better handling of database operations. Without stricter schemas and validation, malformed records could reach the database. Without indexes, common lookups and sorted trip retrieval would be less efficient as data grew.

Design Decisions

I used Mongoose schema options and validators to move data integrity closer to the persistence layer. The trip schema uses `strict: 'throw'`, timestamps, no version key, field length limits, numeric bounds, date validation, safe image path validation, a unique code index, and a compound start/name index. The user schema stores salted password material with `select: false`, excludes hash and salt from JSON/object transforms, and enforces a unique username index.

Implementation Details

The database module now reads `MONGODB_URI`, `MONGODB_TIMEOUT_MS`, `MONGODB_ENSURE_INDEXES`, and `TRAVLR_SEED_DATA` from configuration. It disables command buffering, enables sanitized filters, sets strict query behavior, reuses a connection promise, initializes model indexes, and seeds normalized trip data only when explicitly enabled. The trip service uses lean reads for list and detail endpoints, `countDocuments` for pagination metadata, `findOneAndUpdate` with `runValidators: true`, and `findOneAndDelete` for delete operations. Duplicate keys, validation errors, strict mode errors, and invalid ObjectIds are normalized into clear API responses.

Challenges Encountered

The main challenge was protecting the database layer while keeping the API usable. Trip records needed human-readable codes, but the API also needed to support ObjectId lookup safely. User records needed authentication support without exposing password hashes or salts. Seed data also needed normalization so records without codes could still satisfy the unique code constraint.

Testing Performed

The repository includes tests for duplicate trip codes, validation failures, invalid ObjectId-like identifiers, unsupported fields, unsafe image values, password validation rules, registration without returning hash or salt values, not-found behavior, and protected database mutation routes.

Results

The enhanced database layer now enforces a clearer data contract, protects sensitive user fields, supports indexed trip lookup and sorting, validates updates, and produces predictable API errors for common persistence failures. These changes make the application more scalable and more resilient to malformed input.

Reflection

This enhancement showed that database design is both a correctness concern and a security concern. Strong schemas, indexes, sanitized filters, and validation-backed CRUD operations reduce risk and make the rest of the application easier to maintain.

Skills Demonstrated

Skills demonstrated include MongoDB modeling, Mongoose schema design, index design, validation, password storage safeguards, duplicate handling, safe query construction, environment-based configuration, and persistence-layer testing.

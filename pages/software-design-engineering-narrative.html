Enhancement One

Software Design & Engineering Narrative

This narrative describes the architectural, security, and maintainability improvements made to the CS465 TRAVLR application for the CS499 capstone.

Artifact Overview

The artifact is the TRAVLR travel application, a Node.js and Express application with public HBS-rendered pages, a REST API, an Angular administrative client, and MongoDB persistence through Mongoose. The enhancement focused on moving the application toward a more enterprise-oriented structure without replacing the core product.

Why the Artifact Was Selected

I selected TRAVLR because it already represented a complete full-stack application, but its original structure provided opportunities to improve separation of concerns, route protection, validation, error handling, and maintainability. This made it a strong candidate for demonstrating software engineering growth.

Enhancement Goals

The main goals were to protect administrative write operations, separate controllers from business logic, centralize request validation and response handling, improve project organization, and make the application easier to test and maintain. I also wanted the Angular admin client to align with the backend security model by storing authentication state, attaching authorization headers, and routing unauthenticated users to the login page.

Analysis of the Original Artifact

The original artifact had the core pieces of a travel application, but important concerns were not isolated enough for long-term maintainability. API routes, validation, persistence behavior, and error responses needed clearer boundaries. Administrative create, update, and delete behavior also needed protection so that public API access did not imply unrestricted data modification.

Design Decisions

I introduced a service layer in `app_api/services/auth.service.js` and `app_api/services/trip.service.js` so controllers could focus on HTTP concerns while services handled authentication and trip persistence operations. I kept route definitions explicit in `app_api/routes/index.js`, where read routes remain public and write routes use the authentication middleware. The Express app now exposes `createApp`, which allows the database connector, API router, and server router to be injected during testing.

Implementation Details

The backend now includes registration and login endpoints, HMAC-signed JWT-shaped tokens, Bearer token middleware, centralized JSON helpers, normalized error handling, request logging, request body size limits, CORS configuration, and database availability middleware. Trip controllers now call shared utilities such as `buildTripPayload`, `validateTripPayload`, `normalizeTripIdentifier`, `parsePagination`, and `sendError` instead of duplicating validation and response logic. The Angular client was reorganized into `core`, `home`, `static-pages`, and `trips` areas, with an auth service, auth guard, auth interceptor, trip service, trip list, trip card, and trip form components.

Challenges Encountered

The primary challenge was improving the architecture without changing the purpose of the application. Authentication had to protect mutating operations while keeping public trip reads available. Error responses also needed to remain predictable across validation errors, duplicate keys, missing records, malformed identifiers, and unexpected server failures.

Testing Performed

The repository includes Node test coverage for registration, login, protected create/update/delete operations, duplicate handling, invalid credentials, pagination headers, identifier validation, and not-found responses. It also includes Angular component tests for the trip list loading and error states and the trip form component creation.

Results

The result is a more modular TRAVLR application with clearer backend boundaries, protected administrative API operations, more consistent error handling, and a client-side admin flow that supports authentication-aware behavior. These changes make the application easier to reason about, test, and extend.

Reflection

This enhancement demonstrates growth from building a working application to engineering a maintainable system. The most important improvement was not a single feature, but the separation of responsibilities across routes, controllers, services, middleware, utilities, and Angular feature areas.

Skills Demonstrated

Skills demonstrated include modular architecture, REST API design, route protection, authentication workflow design, validation, centralized error handling, logging, testability, Angular routing, Angular services, HTTP interceptors, and maintainable project organization.
